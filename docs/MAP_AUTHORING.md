# GateGrave Map Authoring Specification

Date: 2026-03-13

Purpose:
- Define the clean authoring pipeline for map assets and compiled map metadata.
- Keep masks easy for humans to create.
- Keep runtime/session state authoritative.
- Avoid hand-coding large coordinate sets.

This document is intentionally about the authoring and compile pipeline.
It does **not** lock in future UI overlay colors for things like movement, range, reach, or spell areas.
Those are runtime render concerns, not source-authoring concerns.

## Core Principle

Maps should follow this rule:

- Humans author maps and masks.
- A compile step converts those into structured metadata.
- Runtime/session/combat systems use the compiled metadata as gameplay truth.
- `apps/map-system` renders and previews from that truth.

So:
- masks are authoring input
- compiled JSON/metadata is runtime truth
- live gameplay should not depend on raw pixel inspection every turn

This document currently gives the most concrete examples for dungeon-map masks because that is the active authoring workflow, but the same author -> compile -> canonical-data model should be used for combat maps too.

## Canonical File Types

### 1. Base map image
Purpose:
- visual background for the map

Typical locations:
- `apps/map-system/assets/base-maps/dungeon/`
- `apps/map-system/assets/base-maps/combat/`

### 2. Terrain/wall mask
Purpose:
- blocked tiles
- edge walls
- difficult terrain
- hazards
- terrain semantics

Typical locations:
- `apps/map-system/assets/masks/dungeon/`
- `apps/map-system/assets/masks/combat/`

Current implementation path:
- shared terrain-mask loader and palettes in `apps/map-system`

### 3. Marker mask
Purpose:
- party start/party position seed
- visible enemy markers
- traps
- exits
- future room/object markers if needed

Typical location:
- `apps/map-system/assets/masks/dungeon/`

Current implementation path:
- `apps/map-system/src/core/dungeon-mask-loader.js`
- `apps/map-system/data/dungeon/dungeon-mask-palettes.json`

### 4. Compiled map/profile data
Purpose:
- the structured form runtime/session/combat/map rendering should actually consume

Typical locations:
- `apps/map-system/data/maps/dungeon/`
- `apps/map-system/data/profiles/dungeon/`
- `apps/map-system/data/maps/combat/`
- `apps/map-system/data/profiles/combat/`

## Runtime Authority Boundary

The authoritative gameplay truth should be:
- compiled map metadata
- room/session/combat state
- content metadata

The authoritative gameplay truth should **not** be:
- raw painted PNG pixels
- runtime re-guessing of image semantics

This keeps the architecture safe:
- gateway stays presentation-only
- session and combat state remain authoritative for progression and position
- map-system remains rendering/preview/selection support

## Current Authoring Workflow

1. Create a clean base map PNG.
2. Create a terrain/wall mask of the same size.
3. Create a marker mask of the same size if the map type needs one.
4. Register the map JSON with paths to those assets.
5. Run the compile/loader step.
6. Produce structured map/profile data.
7. Use that compiled data for rendering and authoritative session/combat-driven display.

## Validation Control Workflow

Before debugging prettier authored maps, validate behavior on clean control maps first.

Recommended control-profile pattern:
- keep one clean combat validation profile and one clean dungeon validation profile for the shared 12x10 test maps
- use them for terrain semantics sanity checks
- use them for hazard/difficult-terrain and edge-wall regression checks
- use them for dungeon marker, overlay, and debug-label validation

Current status:
- this workflow is the recommended direction for map validation
- the clean control-profile artifacts still need to be restored onto `main` before the file paths can be treated as canonical references

Practical use:
- layer the validation profile on top of the compiled base profile for the same map
- confirm the semantics/debug output there first
- only then move on to prettier authored maps when something looks wrong

## Current Dungeon Marker Semantics

These are the current supported dungeon marker meanings:
- red = enemy marker
- yellow = party marker
- blue = trap
- purple = exit

Important:
- these are configurable through the dungeon mask palette file
- they are not the same thing as runtime overlay colors

## What We Do Not Need To Decide Yet

We do **not** need to decide final authored mask colors for:
- movement highlight
- attack range
- reach preview
- spell range
- spell AoE

Reason:
- those should be generated by the renderer from gameplay/session/combat state
- they are not dungeon authoring inputs

So the authoring specification can be finalized now without freezing those future overlay color decisions.

## Compile Output Expectations

The compiled map data should be able to express:
- base map dimensions
- tile size
- blocked tiles
- edge walls
- hazards/difficult terrain
- party start or map-seeded party position
- exits
- traps
- visible enemies / encounter markers
- room/object markers

## Recommended Long-Term Direction

Authoring should become:
- easy in MS Paint or similarly simple tools
- stable for content creators
- deterministic for runtime

The long-term ideal is:
- humans paint masks
- tools compile masks into canonical JSON/metadata
- runtime/session uses that metadata
- map-system renders from the same metadata

## Practical Rule For The Team

When deciding whether something belongs in an authored mask:
- if it is authored world/map truth, it belongs in a mask or compiled metadata
- if it is a temporary player-facing preview, it belongs in runtime rendering logic

Examples:
- wall = authoring truth
- trap marker = authoring truth
- exit marker = authoring truth
- party position seed = authoring truth
- movement highlight = runtime preview
- spell radius highlight = runtime preview
- selected target ring = runtime preview
