1) Architecture Boundary
------------------------
The map system has been kept isolated and intentionally does NOT own authoritative gameplay mutation.

It currently owns:
- map rendering
- token rendering and token selection
- movement preview
- attack preview
- spell target/range/area preview
- terrain and mask authoring helpers
- normalized action contracts for later handoff

It does NOT currently own:
- final combat resolution
- persistence as source of truth
- canonical event execution outside map-side interaction normalization
- gateway logic

This is consistent with project rules:
- event-driven architecture
- gateway contains no gameplay logic
- systems do not directly mutate each other
- combat instances stay isolated
- database remains source of truth


2) Major Completed Work
-----------------------
The following foundations are implemented inside `apps/map-system`:

- canonical map state schema
- grid-based coordinates and movement rules
- default `1 tile = 5 feet`
- alternating diagonal movement rule support
- occupancy rules so only one actor occupies one tile
- no diagonal corner-cutting through blocked terrain
- physical attack range preview
- spell range preview
- spell area-of-effect preview
- spell target-selection interaction flow
- attack target-selection interaction flow
- unified interaction controller for Move / Attack / Spell / Token / Back / Confirm
- normalized handoff action contracts
- player token pipeline
- enemy token pipeline
- token cleanup processing
- token catalog and token selection foundation
- terrain catalog and terrain stamping helpers
- MS Paint mask workflow for terrain authoring
- layered profile support for combining authored terrain with preview/token profiles


3) Important Current Conclusion
-------------------------------
The tile positioning problem is NOT a general renderer failure.

It was tested against clean control maps:
- `map_12x12.png` / `mask-12x12.png`
- `map_12x10.png` / `mask-12x10.png`

On the clean `12x12` map, token placement is correct.

This means:
- the current renderer can place tokens correctly on mathematically aligned maps
- earlier placement issues were caused by art/grid calibration problems in painterly or irregular test maps, especially the retired `forest-road` demo and the first cave test map


4) Current Terrain Authoring Reality
------------------------------------
Tile-based mask authoring works for:
- solid obstacles that clearly occupy tiles
- boulders
- pillars
- pits
- lava / hazards
- tree tiles
- fully occupied obstacle shapes

Tile-based mask authoring does NOT model thin line walls well when the wall lives between tiles.

Important design conclusion:
- thin dungeon walls should eventually use an edge-based wall system
- tile masks remain the right approach for tile-occupying obstacles

This means the next major feature should be edge-wall detection and edge-wall blocking rules for:
- movement
- melee attacks
- ranged attacks
- spell targeting / line of sight


5) MS Paint Mask Workflow
-------------------------
The current palette is:
- `#FFFFFF` = open ground
- `#000000` = wall
- `#FF0000` = impassable terrain
- `#0000FF` = water / impassable
- `#00FF00` = difficult terrain
- `#FFFF00` = tree / blocks sight
- `#FF00FF` = pit / impassable
- `#00FFFF` = cliff / blocks sight

Current workflow:
1. Create map PNG.
2. Create same-size mask PNG.
3. Paint tile centers with flat colors.
4. Run terrain mask conversion.
5. Render previews or play from generated combat profile.

Current limitation:
- mask reading is tile-center based, not edge based

10) Practical Advice For Next Session
-------------------------------------
When testing placement or movement bugs:
- use the clean `12x12` and `12x10` control maps first
- do not debug on painterly maps first

When testing terrain masks:
- trust tile-center fills for tile obstacles
- do not expect thin line walls to work correctly until edge-wall support exists

When rendering previews that depend on authored terrain plus preview tokens:
- use layered profiles
- example pattern:
  `combat-profile.json,preview-profile.json`
